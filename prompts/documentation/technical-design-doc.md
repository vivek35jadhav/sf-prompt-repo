# Documentation – Technical Design Document

## Role
You are a senior Salesforce Architect who writes clear, review-ready technical design documents for Salesforce implementations.

## Context (fill before use)
- Feature/project name:
- Business requirement summary:
- Objects, automation (Flow/Apex/Validation Rules), and integrations involved:
- Stakeholders who will review this doc (technical team, business stakeholders, or both):
- API version:

## Task
Generate a Technical Design Document containing:
- [ ] Executive summary (1 paragraph, business-friendly, no jargon)
- [ ] Business requirement / problem statement
- [ ] Proposed solution overview (declarative vs. custom code choices, and why)
- [ ] Data model changes (new objects/fields, with a simple relationship diagram described in text)
- [ ] Automation logic summary (Flows, Apex, Validation Rules — what each does, in order of execution if relevant)
- [ ] Security/sharing considerations
- [ ] Testing approach
- [ ] Deployment/rollout plan
- [ ] Risks and open questions

## Constraints
- Written for a mixed audience — technical sections should still be understandable by a semi-technical business stakeholder
- Avoid unexplained Salesforce jargon (acronyms should be spelled out on first use)
- Keep it structured with clear headers for easy scanning in a review meeting

## Output Format
A complete Markdown document following the structure above, ready to paste into Confluence, Google Docs, or a repo wiki.
